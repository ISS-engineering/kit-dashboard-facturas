# Locale file schema

Locale files live in `assets/locales/<lang>.json`. Each one provides every translatable string the dashboard renders, plus the formatting locale for numbers and dates.

## Required top-level fields

| Field | Type | Example |
|---|---|---|
| `lang_code` | string (ISO 639-1) | `"en"` |
| `lang_native_name` | string | `"English"` |
| `formatting_locale` | string (BCP 47, used by `babel`) | `"en_US"` |
| `ui` | object | (see below) |

## Placeholder syntax

Strings use Python's `str.format()` syntax: `{name}`. The renderer fills these slots from the metrics JSON. Example: `"{count} invoices analyzed".format(count=16)` → `"16 invoices analyzed"`.

## Number / date / currency formatting

The script uses `babel.numbers.format_currency` and `babel.dates.format_date` with `formatting_locale`. Examples:

| `formatting_locale` | Net 1234.56 EUR | Date 2026-03-20 |
|---|---|---|
| `en_US` | `€1,234.56` | `Mar 20, 2026` |
| `es_ES` | `1.234,56 €` | `20 mar 2026` |
| `de_DE` | `1.234,56 €` | `20. März 2026` |
| `fr_FR` | `1 234,56 €` | `20 mars 2026` |
| `it_IT` | `1.234,56 €` | `20 mar 2026` |

Currency symbol comes from the invoice's `moneda` field, not the locale.

## Adding a new language

1. Copy `en.json` → `<new-code>.json`
2. Translate every string value (keep keys and `{placeholders}` intact)
3. Set `lang_code` and `lang_native_name`
4. Set `formatting_locale` to a valid BCP 47 code supported by `babel`
5. Run the renderer with `--lang <new-code>` and visually inspect the dashboard

No code changes needed.

## Validation

`render-dashboard.py` validates the chosen locale on load:
- All required keys present (compared to `en.json` as canonical)
- `formatting_locale` is recognized by `babel`
- Every `{placeholder}` in the locale file is one the renderer knows how to fill

If validation fails the script exits with an error pointing to the missing/extra keys.
